# tunaRound mesh 아키텍처

이 문서는 여러 머신의 터미널 에이전트와 워커가 어떻게 연결되는지 설명합니다.

명령어를 그대로 따라 실행하려는 경우에는 [A2A 작업 위임 사용법](a2a-usage.md)을 먼저 보는 편이 빠릅니다. 이 문서는 구조와 역할을 이해할 때 사용합니다.

## 한눈에 보기

```text
                    ┌──────────────────────┐
                    │  tunaRound 코어      │
                    │                      │
                    │  작업 큐             │
                    │  공유 전사·검색      │
                    │  로스터·이벤트       │
                    └──────────┬───────────┘
                               │
             ┌─────────────────┼─────────────────┐
             │                 │                 │
      ┌──────▼──────┐   ┌──────▼──────┐   ┌──────▼──────┐
      │ 라이브 세션 │   │ 헤드리스 워커│   │ 외부 A2A    │
      │ Claude/Codex│   │ work / node  │   │ 에이전트     │
      └─────────────┘   └─────────────┘   └─────────────┘
```

한 머신이 코어를 맡고, 나머지 머신의 세션과 워커가 이 코어에 접속합니다.

사람은 한 자리에서 작업을 보내고 결과를 확인합니다. 다른 머신으로 이동해 프롬프트와 결과를 직접 복사하는 일을 줄이는 것이 목적입니다.

## 구성 요소

| 구성 요소 | 실행 형태 | 역할 |
| --- | --- | --- |
| 코어 | `tunaround serve` | 작업 큐, 공유 전사, 검색, A2A API를 제공합니다. |
| 라이브 세션 | Claude Code, Codex 등의 TUI | 사람이 직접 대화하고 판단하는 자리입니다. |
| 워커 | `tunaround work` | 작업을 가져가 러너로 처리하고 결과를 반환합니다. |
| 노드 | `tunaround node` | 한 머신의 브로커·워커 구성을 설정대로 상주시킵니다. |
| presence 스캐너 | `presence-scan` | 로컬에서 실행 중인 에이전트 세션을 찾아 로스터에 반영합니다. |
| Codex relay | `codex-relay` | Codex 라이브 세션에 외부 작업을 대신 주입합니다. |
| 결과 감시기 | `watch-results` | 완료·실패한 작업을 실시간으로 알려줍니다. |

대부분의 구성 요소는 작업이 없을 때 LLM을 호출하지 않습니다. 데몬이 떠 있다고 해서 유휴 상태에서 토큰을 계속 소비하지는 않습니다.

## 코어가 맡는 일

코어는 mesh의 공용 상태를 보관합니다.

- 작업과 상태 전이
- 세션 전사
- 검색 색인
- 에이전트와 워커 로스터
- 대시보드 이벤트
- 완료된 작업 결과
- mesh 토론 오케스트레이션(`start_discussion`이 만든 토론의 라운드 진행·종합, 전사는 `debate:<id>` 세션. `gate=true`면 라운드 사이에 사람 승인 대기)

작업의 현재 상태에 대한 권위는 코어 하나에 있습니다. 여러 워커가 같은 작업을 동시에 처리하지 않도록 claim과 lease도 코어가 관리합니다.

## 세션과 워커의 차이

### 라이브 세션

사람이 직접 보고 대화하는 터미널입니다.

설계 검토, 진단, 방향 결정처럼 사람과 계속 상호작용해야 하는 작업에 적합합니다.

### 헤드리스 워커

사람이 붙어 있지 않아도 정해진 작업을 가져가 실행합니다.

```text
poll → claim → runner 실행 → complete 또는 fail
```

구현, 테스트, 문서 정리처럼 입력과 종료 조건이 비교적 분명한 작업에 적합합니다.

## 작업이 처리되는 흐름

```text
사람 또는 dispatcher
        │
        │ 작업 생성
        ▼
     submitted
        │
        │ 워커가 선점
        ▼
      working
        │
        ├──────────────▶ completed
        ├──────────────▶ failed
        ├──────────────▶ canceled
        └──────────────▶ input_required
```

기본 흐름은 다음과 같습니다.

1. 사람이 목표를 작성합니다.
2. 코어가 작업을 `submitted` 상태로 저장합니다.
3. 대상 워커가 작업을 발견하고 claim합니다.
4. 워커가 Claude, Codex, OpenCode, HTTP LLM 또는 외부 A2A 에이전트로 작업을 실행합니다.
5. 결과와 산출물을 코어에 반환합니다.
6. 사람은 결과를 확인하고 채택, 수정, 재작업 여부를 결정합니다.

작업은 특정 에이전트 ID로 보낼 수도 있고, 태그 조건으로 적합한 워커를 찾을 수도 있습니다.

## 에이전트 ID와 태그

두 값의 역할은 다릅니다.

| 값 | 역할 | 예시 |
| --- | --- | --- |
| 에이전트 ID | 실제 작업을 전달할 대상 | `mac-codex-worker` |
| 태그 | 조건에 맞는 대상을 찾기 위한 정보 | `machine=mac,runner=codex,role=worker` |

태그로 대상을 찾더라도 최종 작업에는 구체적인 에이전트 ID가 기록됩니다.

권장 태그 키는 다음과 같습니다.

- `machine`: `win`, `mac`, `linux`
- `runner`: `claude`, `codex`, `opencode`, `http`, `a2a`
- `role`: `session`, `worker`, `infra`
- `project`: 작업 대상 저장소
- `mode`: `read`, `write`

작업 대상은 실제로 poll하는 세션이나 워커여야 합니다. 작업을 보내기만 하는 dispatcher를 대상으로 지정하면 작업이 소비되지 않습니다.

## 작업이 멈췄을 때

### 소비자가 없는 작업

오래 `submitted` 상태에 머무르면 `⚠no-consumer?` 표시가 붙습니다.

보통 다음 원인 중 하나입니다.

- 대상 에이전트 ID 오타
- 워커가 실행되지 않음
- 태그 조건에 맞는 워커가 없음
- dispatcher를 대상으로 잘못 지정함

### 실행 중 멈춘 작업

오래 갱신되지 않는 `working` 작업에는 `⚠stuck?` 표시가 붙습니다.

워커가 비정상 종료된 경우 lease가 만료되면 작업을 다시 `submitted`로 돌릴 수 있습니다. 재시도 횟수가 상한을 넘으면 무한 반복하지 않고 `failed`로 종료합니다.

## 여러 에이전트가 같은 저장소를 수정할 때

mesh는 작업 전달을 조정하지만 Git 충돌 자체를 없애지는 않습니다. 병렬 작업에서는 다음 규칙을 지키는 편이 안전합니다.

1. 의미 있는 변경은 별도 브랜치에서 작업합니다.
2. 한 브랜치나 worktree는 한 에이전트만 사용합니다.
3. 작업을 시작하면 A2A task를 claim해 담당자를 명확히 합니다.
4. 변경은 오래 미커밋 상태로 두지 않습니다.
5. 공유 파일은 통합 담당자가 최종 반영합니다.

역할을 나눈다면 다음 정도면 충분합니다.

| 역할 | 책임 |
| --- | --- |
| 통합자 | 방향 결정, 리뷰, main 반영 |
| 감독 세션 | 진단, 설계 검토, 작업 분해 |
| 워커 | 할당된 브랜치에서 구현하고 결과 보고 |

이 역할은 프로토콜이 강제하는 권한 체계가 아니라 협업 충돌을 줄이기 위한 운영 관례입니다.

## 상태의 영속과 재생

코어를 재시작하거나 대시보드를 새로 열어도 작업 기록은 SQLite에 남습니다.

- 작업과 결과는 `tasks` 장부에서 복원됩니다.
- 대시보드는 최근 작업 상태를 다시 받아 화면을 구성합니다.
- 결과 감시기는 연결이 끊겼다가 복구되어도 놓친 완료·실패 이벤트를 다시 받을 수 있습니다.
- 완료된 작업의 요청과 결과는 별도 네임스페이스로 색인되어 이후 검색할 수 있습니다.

휘발성 이벤트 스트림은 알림용이며, 작업 기록의 기준은 SQLite입니다.

## 자율화 범위

현재 구조는 **semi-A2A**, 즉 사람을 포함한 작업 흐름입니다.

자동화되는 부분:

- 대상 발견
- 작업 claim
- 러너 실행
- 완료·실패 보고
- 결과 알림

사람이 맡는 부분:

- 목표 결정
- 쓰기 권한 부여
- 결과 채택
- 재작업과 취소 판단
- 최종 통합

사람 없이 에이전트끼리 목표를 계속 만들어내는 무한 자율 루프는 두지 않습니다.

## A2A 표준 호환 범위

tunaRound는 A2A 프로토콜의 구조를 사용하지만, 코어 내부에는 자체 라우팅 확장이 있습니다.

- **Outbound**: tunaRound가 외부 표준 A2A 에이전트에 작업을 보내고 결과를 회수하는 경로는 지원합니다.
- **Inbound**: 임의의 제3자 A2A 클라이언트가 tunaRound 코어에 바로 작업을 넣는 완전 호환 서버는 목표가 아닙니다.

외부 에이전트와의 완전한 양방향 호환이 필요해지면 표준 A2A와 tunaRound 작업 큐 사이에 별도 어댑터가 필요합니다.
