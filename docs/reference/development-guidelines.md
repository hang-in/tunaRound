# 개발 규칙

이 문서는 tunaRound에 코드를 추가하거나 수정할 때 지켜야 할 기본 규칙입니다.

목표는 복잡한 절차를 만드는 것이 아니라 다음 세 가지를 지키는 것입니다.

1. 요청한 범위만 바꿉니다.
2. 구현 전에 경계와 성공 조건을 확인합니다.
3. 완료라고 말하기 전에 실제 검증을 끝냅니다.

작은 오타 수정처럼 명백한 작업에는 전체 절차를 기계적으로 적용하지 않아도 됩니다.

## 1. 구현 전에 확인할 것

코드를 쓰기 전에 다음 질문에 답할 수 있어야 합니다.

- 무엇을 바꾸는가
- 무엇은 바꾸지 않는가
- 성공 여부를 어떤 테스트나 명령으로 확인하는가
- 해석이 여러 가지라면 어느 쪽을 선택했고 왜 그런가

불확실한 부분을 추측으로 채우지 않습니다. 실제 코드, 로그, 테스트 결과를 먼저 확인합니다.

비단순 작업은 짧은 계획을 남깁니다.

```text
1. 현재 동작 재현
2. 변경 범위 확인
3. 최소 수정
4. 회귀 테스트
```

장기 작업이나 여러 세션이 이어받는 작업은 plan, checklist, context notes를 사용합니다.

| 문서 | 역할 |
| --- | --- |
| plan | 무엇을 왜 바꾸는지 설명 |
| `checklist.md` | 실행할 작업과 진행 상태 기록 |
| `context-notes.md` | 작업 중 결정과 근거 기록 |

## 2. 가장 단순한 해결책을 선택하기

문제를 해결하는 데 필요한 최소 코드만 추가합니다.

다음은 기본적으로 피합니다.

- 요청하지 않은 기능
- 한 번만 쓰는 추상화
- 필요성이 확인되지 않은 설정 옵션
- 실제로 발생하지 않는 상황을 위한 복잡한 방어 코드
- 미래 확장을 이유로 미리 만든 훅과 인터페이스

확장성은 미사용 코드를 미리 만드는 데서 나오지 않습니다. 현재 책임이 명확하고 의존 방향이 깨끗하면 이후 확장이 쉬워집니다.

## 3. 변경 범위를 좁게 유지하기

현재 작업과 직접 관련된 파일과 줄만 수정합니다.

- 지나가다 발견한 포맷이나 주석을 함께 고치지 않습니다.
- 요청과 무관한 리팩터링을 섞지 않습니다.
- 기존 스타일을 따릅니다.
- 이번 변경으로 새로 생긴 미사용 import, 변수, 함수는 제거합니다.
- 기존 dead code는 별도 문제로 남기고 현재 diff에 섞지 않습니다.

커밋 전에 diff를 보고 각 변경이 요청과 직접 연결되는지 확인합니다.

## 4. 도메인과 경계

새로운 도메인 개념은 문자열 관례보다 Rust 타입으로 표현합니다.

예를 들어 역할, 작업 상태, 메시지 종류처럼 가능한 값이 정해져 있다면 `enum`이나 전용 타입을 우선합니다.

코어 의존 방향은 안쪽으로 향해야 합니다.

```text
프런트엔드·CLI·HTTP
        ↓
응용 계층·오케스트레이션
        ↓
도메인 타입·순수 로직
```

코어 로직이 REPL, 대시보드, 특정 transport를 직접 import하지 않도록 합니다.

분기, 파싱, 계산, 프롬프트 조립은 가능하면 순수 함수로 분리합니다. 순수 함수는 상태 없이 입력과 출력만 가지므로 테스트와 교체가 쉽습니다.

## 5. 횡단 변경은 먼저 접점을 찾기

메시지 ID, 세션, 작업 상태, 인증처럼 여러 계층을 건드리는 개념은 수정 전에 접점을 적습니다.

```text
타입 → 저장소 → API → CLI/MCP → 대시보드 → 테스트 → 문서
```

한 계층만 바꾸고 다른 계층을 빠뜨리는 일을 막기 위한 절차입니다.

## 6. 파일과 모듈

한 모듈은 한 가지 책임을 갖습니다.

다음은 분리를 검토할 신호입니다.

- 서로 다른 관심사가 한 파일에 섞임
- 같은 변경에서 파일의 여러 구역을 계속 왕복함
- 테스트하려면 큰 상태를 함께 준비해야 함
- 이름만으로 파일의 책임을 설명하기 어려움

줄 수만으로 자동 분리하지는 않습니다. 책임이 하나라면 긴 파일도 가능하고, 책임이 둘이라면 짧아도 분리 대상입니다.

새 소스 파일에는 첫 줄에 역할을 설명하는 짧은 한국어 주석을 둡니다.

```rust
// 토론 라운드에 전달할 프롬프트를 조립합니다.
```

설정 파일과 데이터 파일에는 적용하지 않아도 됩니다.

## 7. 오류를 추측하지 않기

실패하면 먼저 실제 오류와 로그를 읽습니다.

1. 전체 오류 메시지와 스택을 확인합니다.
2. 재현 조건을 고정합니다.
3. 필요한 경우 로그를 추가해 상태를 확인합니다.
4. 원인이 확인된 뒤 수정합니다.
5. 같은 재현 절차로 다시 검증합니다.

기억에 의존해 흔한 해결책부터 적용하지 않습니다.

## 8. 테스트와 검증

버그 수정은 가능하면 실패하는 재현 테스트부터 만듭니다.

기능 변경은 성공 조건이 드러나는 테스트를 함께 추가합니다.

완료 전 기본 검증입니다.

```bash
cargo fmt -- --check
cargo test
cargo clippy --all-targets --all-features -- -D warnings
```

변경 범위가 작을 때는 관련 테스트부터 실행해 빠르게 확인하고, 마지막에는 필요한 전체 검증을 실행합니다.

외부 서버나 플랫폼이 필요한 검증을 실행하지 못했다면 완료 보고에 명시합니다. 실행하지 않은 테스트를 통과했다고 쓰지 않습니다.

## 9. 커밋

한 커밋에는 한 논리 변경만 담습니다.

좋은 기준은 커밋을 한 문장으로 설명할 수 있는지입니다.

```text
fix: reject stale task completion
feat: add selector routing to worker registry
docs: rewrite source build guide
```

기능과 무관한 포맷 수정, 문서 개편, 리팩터링을 한 커밋에 섞지 않습니다.

커밋 전에 검증하고, 커밋 후 push 전 현재 브랜치와 원격 상태를 다시 확인합니다.

## 10. 병렬 작업

여러 에이전트가 같은 레포에서 일할 때는 다음 규칙을 지킵니다.

1. 비단순 변경은 브랜치에서 처리합니다.
2. 한 브랜치는 한 에이전트만 편집합니다.
3. 병렬 작업은 worktree 또는 별도 클론으로 분리합니다.
4. 공유 파일은 총괄 한 곳에서 병합합니다.
5. 작업 범위와 완료 조건을 task 본문에 적습니다.
6. 완료 결과에는 변경 내용, 검증 결과, 남은 문제를 포함합니다.

## 11. 커밋 전 확인표

- [ ] 요청한 범위만 변경했는가
- [ ] 새 개념을 적절한 타입으로 표현했는가
- [ ] 코어가 프런트엔드나 transport에 불필요하게 의존하지 않는가
- [ ] 분기와 계산을 테스트 가능한 함수로 분리했는가
- [ ] 미사용 추상화나 설정을 미리 만들지 않았는가
- [ ] 오류 원인을 실제 로그와 재현으로 확인했는가
- [ ] 관련 테스트와 전체 필수 검증을 실행했는가
- [ ] 문서와 예제가 현재 동작과 맞는가
- [ ] 커밋이 하나의 논리 변경으로 설명되는가

이 문서는 사람이 읽는 기여 규칙의 정본입니다. 에이전트별 지침 파일에는 필요한 핵심 규칙만 요약하고, 세부 내용은 이 문서를 참조합니다.
