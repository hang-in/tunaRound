## 0. 재작성이 필요한 이유 (맥락)

현재 README는 제품의 정본이 아닌 경로를 첫 화면에 놓고 있다.

- **정본**: 총감독(사람)이 앉은 외부 Claude Code 세션에 tuna-broker MCP를 붙이고, 거기서 로컬/LAN의 다른 에이전트(Codex 등)에게 A2A로 위임하고 결과를 받는 것.
- **현재 README 첫 화면**: `tunaround chat` REPL을 "가장 빠르게 시작하기"로 제시. 그러나 REPL은 2-에이전트 토론 도구일 뿐이며 task 코드가 0건이다. 정본이 아니다.

즉 제품의 진짜 진입 경로(외부 Claude Code에 MCP 등록)가 README에 없고, reference 문서(a2a-usage.md)에만 숨어 있다.

**재작성의 목표**: 첫 화면이 정본을 가리키게 한다. 나머지는 전부 아래로 내린다.

## 1. 첫 성공의 정의 (README 전체가 이것을 향한다)

신규 사용자가 도달해야 할 첫 성공은 단 하나다.

> 총감독 Claude Code 세션에서 로컬 Codex에게 대화를 보내고, 응답을 받는 것.

- LAN/원격, 검색, 대시보드, 워커 노드, REPL은 첫 성공이 **아니다**. 전부 "다음 단계".
- 사용자는 코어/워커/브로커/토큰/A2A라는 단어를 **몰라도** 첫 성공에 도달해야 한다.
- README 첫 화면에 남을 어휘는 `init`, `node`, 그리고 "Claude Code 재시작"뿐이다.

## 2. 전제: 온보딩 구현 상태 (README는 구현을 앞서가면 안 된다)

README를 쓰기 전에 온보딩 감사(v2-54)의 P0 구현 여부를 확인할 것.

| P0 항목 | 내용 | README 서술 조건 |
|---|---|---|
| P0-① 로컬 프리셋 | init 기본 = 127.0.0.1 + 무토큰 | 구현됐으면 토큰 어휘를 첫 화면에서 완전히 제거 |
| P0-② MCP 자동 등록 | init이 `claude mcp add`를 멱등 래핑 + 재시작 안내 출력 | 구현됐으면 수동 `claude mcp add`를 첫 화면에서 제거 |
| P0-③ 러너 전수 탐지 | claude·codex·opencode 전부 lane 스캐폴드 | 구현됐으면 `--runner codex` 플래그를 첫 화면에서 제거 |

**중요**: P0가 아직 구현 전이면, README가 자동화된 척 쓰면 안 된다. 구현 전이라면 현재 실제 명령(수동 `claude mcp add` 포함)을 정직하게 쓰되, "곧 자동화될 예정"이라고 미래를 약속하지도 말 것. 지금 되는 것만 쓴다.

이 문서의 예시는 **P0 구현 완료 상태**를 가정한 것이다. 구현 상태에 맞춰 명령 줄을 조정할 것.

## 3. 새 README 구조 (섹션 순서가 곧 우선순위다)

아래 순서를 지킬 것. 위에 있을수록 정본에 가깝고, 아래로 갈수록 이스케이프 해치다.

### 3.1 첫 문장 / About

기능 나열이 아니라, tunaRound가 **없애주는 통증**을 먼저 말한다.

첫 문장 후보(P0 구현 완료 기준):

> 쓰던 Claude Code에 tunaRound를 붙이면, 거기서 Codex와 다른 에이전트에게 일을 시키고 결과를 받습니다. 터미널을 옮겨다니며 복사·붙여넣기 하지 않아도 됩니다.

- 핵심어는 "복붙하지 않아도 된다". 이것이 통증이다.
- "한곳에서 보고, 대화하고, 작업을 맡기고, 결과를 모으는" 같은 기능 나열형 문장으로 시작하지 말 것. 그건 tunaRound 안으로 들어오라는 초대이고, 정본과 반대 메시지다.
- 사람이 최종 판단을 쥔다는 점은 유지하되(자율 swarm 아님), 첫 문장의 주어는 "당신(총감독)"이지 "tunaRound"가 아니다.

⚠ 주의: "이미 열려 있는 TUI 세션에 그대로 붙는다"고 쓰지 말 것. 코드 조사 결과, 기본 경로는 헤드리스 fresh spawn이고 이미 열린 claude TUI에는 직접 주입이 안 된다(codex도 --remote attach 조건부). 첫 성공은 헤드리스 응답이다. "이미 열어둔 터미널을 그대로 두세요"는 총감독 세션에만 참이고 워커에는 거짓이므로, 사용자를 오도한다. "쓰던 Claude Code에 붙인다"까지가 정확한 표현이다.

### 3.2 첫 성공 (Quick Start)

명령은 최소로. P0 구현 완료 기준 이상적 형태:

```bash
# 설치 (한 줄)
curl ... | sh   # 또는 brew / irm

# 초기화 + 기동
tunaround init
tunaround node

# Claude Code 재시작 (필수 - 아래 설명)

# 총감독 Claude Code 세션에서:
> codex한테 이 설계 검토받아줘
→ 응답 도착
```

**재시작을 반드시 명시할 것.** MCP 서버는 실행 중인 Claude Code 세션에 핫 리로드되지 않는다(공식 문서상 부재). init 출력의 마지막 줄에도 재시작 안내가 있어야 하지만, README에서도 이 단계를 생략하거나 흐리지 말 것. 숨기면 사용자는 "안 되는데" 하고 나간다.

이 섹션에 코어/워커/브로커/토큰/A2A/serve/join 같은 단어가 등장하면 안 된다.

### 3.3 무엇을 할 수 있나

현재 README의 "무엇을 할 수 있나" 표는 대체로 살릴 수 있으나, **배송 불일치를 수정할 것**:

- 현재 표는 "로컬과 LAN의 세션을 한 화면에서 관찰합니다"를 약속한다.
- 그러나 릴리스 바이너리에는 대시보드 SPA가 포함되지 않는다(소스 빌드 필요).
- → 대시보드가 첫 성공/기본 경험이 아님을 분명히 하거나, 이 항목을 "다음 단계"로 옮길 것. 설치 스크립트로 들어온 사용자가 도달할 수 없는 기능을 첫 화면에서 약속하지 말 것.

### 3.4 명령 (재구성)

현재는 chat·core·serve·join·init·doctor·node·work·reindex 9개가 **동등하게** 나열돼 있다. 실제 위계는 이렇다:

- **첫 성공에 필요**: `init`, `node`
- **실패 시에만**: `doctor`
- **이스케이프 해치**: `core`, `serve`, `join`, `work`, `reindex`, `chat`

표를 위계에 맞게 재편할 것:
- 상단에 `init` / `node`만.
- 나머지는 "고급 사용" 또는 접기(details) 블록으로 분리.
- `chat`은 여기서 "설계 토론 REPL(부수 기능)"로 명확히 강등. 첫 화면에서 제거.

### 3.5 다음 단계 (첫 성공 이후에만)

첫 성공을 지난 사용자가 읽을 것들. 순서대로 깊어진다:

1. LAN/원격 머신 연결 (join, 토큰, 네트워크)
2. 워커 노드 상주
3. 검색과 기억 (색인, 형태소, 하이브리드 검색)
4. 웹 대시보드 (소스 빌드 필요 명시)
5. 설계 토론 REPL (`tunaround chat`)

각 항목은 요약 2~3줄 + 상세 문서 링크. 본문에서 절차를 다 풀지 말 것.

### 3.6 왜 만들었나 / 아키텍처 / 문서 링크

현재 README의 "왜 만들었나"는 유지 가능. 단, 앞의 첫 문장과 통증(복붙)이 일관되게 이어지도록 다듬을 것. 아키텍처·문서 링크 섹션은 대체로 그대로 둔다.

## 4. 두 목적의 분리 (표면 배치 원칙)

tunaRound에는 두 목적이 있고, 서로 당긴다:

| 목적 | 요구 | 봉사하는 표면 |
|---|---|---|
| 사람이 쉽게 들어오게 | 구조 은폐, 좋은 기본값 | Quick Start, init/node, 첫 성공 |
| 설계 인사이트 공유 | 구조 노출, 의도 설명 | 아키텍처 문서, 설계 기록, 블로그 |

**README는 전자에 봉사한다.** 아키텍처와 설계 의도는 링크로 연결하되 첫 화면에 쏟지 말 것. 읽으러 온 사람에게는 전부 열려 있어야 하지만(문서 링크), 쓰러 온 사람의 경로에 그것들이 끼어들면 안 된다. "안 물어보되, 열면 전부 있다"를 문서 구조로 구현하는 것.
