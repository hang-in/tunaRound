# 코덱스로부터 1차 리팩토링 제안

## 검토 범위와 제약

- 프로젝트 전체 Rust 소스와 관련 테스트·설계 문서를 정적으로 검토했다.
- 코드는 수정하지 않았다.
- 빌드, 테스트, 타입 체크는 실행하지 않았다. 따라서 전체 기능 정상 여부는 단정하지 않는다.
- 검토 도중 `Cargo.lock`, `src/main.rs`, `src/runner/a2a.rs` 등이 외부에서 추가 변경됐다. 문서는 해당 변경이 반영된 최종 작업 트리를 기준으로 작성했다.

## [높음] MCP 실패가 성공 응답으로 전달된다

- 위치: `src/mcp.rs:422`, `src/mcp.rs:426`, `src/mcp_client.rs:158`, `src/worker.rs:171`
- 분류: 기능 / 결합 강도 / 변동성 정렬
- 결합 태그: Functional
- 문제: `claim_task`, `complete_task`, `fail_task` 내부 실패가 `"착수 실패: ..."` 같은 텍스트로 바뀐 뒤 `CallToolResult::success`로 반환된다. 클라이언트는 `isError`만 검사하므로 이를 성공으로 처리한다.
- 왜 문제인가: claim이 실패해도 워커가 러너를 실행하며, complete가 실패해도 `complete 완료`를 출력한다.
- 변경 전파 범위:
  - 이 파일이 바뀌면 같이 바뀔 가능성이 있는 파일: `src/mcp.rs`, `src/mcp_client.rs`, `src/worker.rs`, 관련 MCP 테스트.
  - 전파 원인: 오류 의미가 MCP wire 응답과 자연어 본문에 이중으로 존재한다.
  - 결합 유형: Functional.
- 제안: 실패는 `CallToolResult::error` 또는 `McpError`로 전달하고, 클라이언트 래퍼는 자연어가 아닌 구조화된 성공 결과를 반환하게 한다.
- 리스크: 변경 시 기존 테스트와 대화형 MCP 소비자가 오류 본문을 기대하는지 확인해야 한다.
- 우선순위 근거: 작업 실행·완료 여부를 반대로 판단할 수 있다.

## [높음] A2A 상태 전이가 무조건 UPDATE라 중복 실행과 종료 상태 덮어쓰기가 가능하다

- 위치: `src/store/sqlite.rs:877`, `src/store/sqlite.rs:901`, `src/mcp.rs:197`, `src/a2a_server.rs:122`
- 분류: 기능 / 결합 강도 / 책임
- 결합 태그: Functional
- 문제: 상태 전이에 현재 상태 조건이 없다. 두 워커가 같은 task를 모두 claim할 수 있고, canceled task를 completed로 바꾸거나 completed task를 canceled로 바꿀 수 있다.
- 왜 문제인가: `TaskState` enum은 있지만 허용 전이 규칙은 저장소에 구현되지 않고 호출자에 분산돼 있다.
- 변경 전파 범위:
  - 이 파일이 바뀌면 같이 바뀔 가능성이 있는 파일: `src/store/sqlite.rs`, `src/mcp.rs`, `src/a2a_server.rs`, `src/worker.rs`.
  - 전파 원인: 상태 머신 규칙이 명시적 계약이 아니라 여러 handler의 호출 순서에 의존한다.
  - 결합 유형: Functional.
- 제안: 저장소에 `claim_task`, `cancel_task`, `fail_task`, `complete_task`를 조건부 UPDATE로 구현하고 영향 행 수가 1이 아니면 전이 충돌을 반환한다.
- 리스크: 기존에 허용되던 비정상 전이가 오류로 바뀐다.
- 우선순위 근거: 중복 외부 작업과 결과 덮어쓰기가 발생할 수 있다.

## [높음] 비기본 세션의 Pull 컨텍스트가 `default` 세션을 읽을 수 있다

- 위치: `src/orchestrator/prompt.rs:48`, `src/roster.rs:69`, `src/main.rs:712`, `src/mcp.rs:625`

- 왜 문제인가: `--session other --pull-context --roster ...` 또는 HTTP core/join 경로에서 인자를 생략한 `read_transcript`가 `default`를 조회할 수 있다.
- 변경 전파 범위:

  - 전파 원인: 세션 식별자가 오케스트레이션 계약에 없고 배선 계층이 암묵적으로 전달한다.
  - 결합 유형: Functional.
- 제안: 현재 세션 ID를 `RoundInput`/`PromptContext`의 명시적 필드로 두고 프롬프트와 MCP 기본 세션 배선에서 동일 값을 사용한다.
- 리스크: 프롬프트 스냅샷 및 runner 인자 테스트가 함께 바뀐다.
- 우선순위 근거: 다른 토론 전사를 읽는 기능 오류다.

## [높음] `CoreSync` 모드에서 checkout과 라운드 저장이 일관되지 않다

- 위치: `src/orchestrator/mod.rs:114`, `src/store/retriever.rs:337`, `src/repl/mod.rs:259`, `src/repl/mod.rs:400`, `src/repl/mod.rs:615`
- 분류: 기능 / 경계와 거리 / 책임
- 결합 태그: Intrusive
- 문제:
  - `/checkout`은 인메모리 `head`만 변경한다. 다음 `step` 시작 시 DB 상태를 다시 adopt하므로 checkout이 취소된다.
  - `load_session` 오류는 `None`으로 축소된다.
  - 한 라운드의 발언을 개별 append하며 중간 실패는 로그만 남긴다. 일부만 저장돼도 전체 라운드가 성공 출력된다.
- 왜 문제인가: DB 권위 모드인데 head 변경과 라운드 단위 저장이 DB 계약에 포함되지 않았다.
- 변경 전파 범위:
  - 이 파일이 바뀌면 같이 바뀔 가능성이 있는 파일: `src/orchestrator/mod.rs`, `src/store/retriever.rs`, `src/store/sqlite.rs`, `src/repl/mod.rs`.
  - 전파 원인: `CoreSync`가 저장소 내부 모델을 노출하면서 필요한 원자적 명령은 제공하지 않는다.
  - 결합 유형: Intrusive.
- 제안: `load_session -> Result<Option<_>>`, `set_head`, `append_round` 같은 명시적 계약을 제공하고 라운드 저장을 하나의 트랜잭션으로 처리한다.
- 리스크: core 모드의 저장 흐름이 바뀌므로 기존 DB와 동시 writer 검증이 필요하다.
- 우선순위 근거: 현재 노출된 분기 기능과 전사 일관성이 직접 깨진다.

## [높음] 잘못된 `--context-map`이 쓰기 작업을 다른 레포로 보낼 수 있다

- 위치: `src/main.rs:639`, `src/worker.rs:111`, `src/worker.rs:176`

- 결합 태그: Contract

- 왜 문제인가: 오타가 설정 오류가 아니라 정상 fallback으로 처리된다. `--write`에서는 기본 레포를 수정할 수 있다.

  - 결합 유형: Contract.
- 제안: `parse_context_map() -> Result<HashMap<...>, String>`으로 분리하고 빈 key/value 및 형식 오류를 진입 시 거부한다.

## [높음] watchdog가 프로세스 트리가 아닌 부모 PID만 종료한다

- 위치: `src/runner/exec.rs:119`, `src/runner/exec.rs:184`
- 분류: 기능 / 경계와 거리
- 결합 태그: Contract
- 문제: Unix는 `kill -9 PID`, Windows는 `taskkill /F /PID PID`만 실행한다.
- 왜 문제인가: CLI가 자식 프로세스를 만들면 후손이 살아남을 수 있다. 후손이 stdout/stderr pipe를 유지하면 현재 reader도 종료되지 않을 수 있다.
- 변경 전파 범위:
  - 이 파일이 바뀌면 같이 바뀔 가능성이 있는 파일: `src/runner/exec.rs`, runner timeout 테스트.
  - 전파 원인: OS 프로세스 생명주기 규칙이 공통 실행 helper에 집중돼 있다.
  - 결합 유형: Contract.
- 제안: Windows는 `/T`, Unix는 별도 process group을 생성하고 그룹 전체를 종료한다.
- 리스크: 플랫폼별 프로세스 생성·종료 검증이 필요하다.
- 우선순위 근거: timeout 이후에도 에이전트 작업이 계속 실행될 수 있다.

- 왜 문제인가: 토크나이저 실패나 feature 조합에 따라 같은 질의의 검색 결과가 실행 모드별로 달라진다.

  - 결합 유형: Functional.
- 제안: 색인용/질의용 tokenizer builder를 검색 모듈에 각각 한 번만 구현한다.
- 리스크: 기존 fallback 검색 순위가 달라질 수 있다.
- 우선순위 근거: 장애 가능성은 제한적이지만 변경 누락이 이미 발생했다.

## [중간] 세션 전량 교체가 벡터·유효성 보조행을 정리하지 않는다

- 위치: `src/store/sqlite.rs:289`, `src/store/sqlite.rs:724`
- 분류: 기능 / 책임
- 결합 태그: Model

- 왜 문제인가: orphan 벡터가 top-K를 차지한 뒤 원문 조회에서 버려질 수 있고, 나중에 같은 ID가 재사용되면 이전 유효성 상태를 상속할 수 있다.
- 변경 전파 범위:
  - 이 파일이 바뀌면 같이 바뀔 가능성이 있는 파일: `src/store/sqlite.rs`, reindex 및 persistence 테스트.
  - 전파 원인: 한 메시지의 파생 데이터 수명주기가 세 테이블에 분산돼 있다.
  - 결합 유형: Model.
- 제안: `save_session` 트랜잭션에서 실제 존재하지 않는 ID의 벡터·유효성 행을 함께 정리하거나 FK 정책을 명시한다.
- 리스크: 유효성 메타 보존 정책을 먼저 확정해야 한다.
- 우선순위 근거: 일반 append 경로에서는 드물지만 seed/축소 복원 시 누적된다.

- 우선순위 근거: 장애를 숨기지만 직접 데이터 손상 가능성은 제한적이다.

## [중간] `main.rs`와 `mcp.rs`가 서로 다른 변동 이유를 동시에 가진다

- 위치: `src/main.rs:223`, `src/mcp.rs:105`, `src/mcp.rs:284`, `src/mcp.rs:570`

  - `mcp.rs`가 검색·전사·A2A task 도구와 HTTP 인증·서버 조립까지 담당한다.
- 왜 문제인가: 현재 A2A runner 추가도 CLI 타입, feature 분기, runner factory를 모두 직접 수정하고 있다. 검색 정책 중복도 이 구조에서 발생했다.
- 변경 전파 범위:
  - 이 파일이 바뀌면 같이 바뀔 가능성이 있는 파일: 신규 subcommand/runner/tool마다 `src/main.rs`, `src/mcp.rs`, feature 설정과 테스트.
  - 전파 원인: 자주 바뀌는 통합 코드가 단일 함수·서버 타입에 집중돼 있다.
  - 결합 유형: Functional.
- 제안: 마이크로서비스나 새 레이어 없이 `run_chat`, `run_work`, `run_reindex`, `build_search_backends`와 검색/전사/task tool handler 모듈 정도로만 분리한다.
- 리스크: 기계적 이동 중 feature gate 누락 가능성이 있다.
- 우선순위 근거: 변경 비용 감소 효과는 있으나 즉시 기능 장애는 아니다.

## 문제 없음으로 판단한 항목

- `run_round`와 prompt 조립의 방향성은 문제 없음. 코어가 REPL/HTTP를 import하지 않으며 순수 prompt 함수가 분리돼 있다.
- Claude/Codex/OpenCode stream 파서는 문제 없음. 외부 포맷 파싱이 runner 내부에 국소화돼 있다.
- 광범위한 production `unwrap`/`panic` 패턴은 문제 없음. 논리적으로 보호된 `unwrap` 외에 반복적인 panic 패턴은 확인되지 않았다.
- 파일 크기만을 근거로 한 분리 필요성은 문제로 판단하지 않았다.

## 가장 시급한 리팩토링 3가지

| 순위 | 항목 | 이유 | 예상 수정 범위 |
|---|---|---|---|
| 1 | A2A 상태 머신과 MCP 오류 계약 통합 | 실패를 성공으로 처리하고 중복 claim·종료 상태 덮어쓰기가 가능함 | `store/sqlite.rs`, `mcp.rs`, `mcp_client.rs`, `worker.rs`, `a2a_server.rs` |
| 2 | 세션 ID를 Pull 컨텍스트의 명시적 계약으로 승격 | 비기본 세션에서 다른 전사를 읽을 수 있음 | `orchestrator`, `prompt`, `roster`, `main`, `mcp` |
| 3 | `CoreSync`에 원자적 head/round/error 계약 추가 | checkout 무효화와 부분 저장이 발생함 | `orchestrator`, `repl`, `store/retriever`, `store/sqlite` |
