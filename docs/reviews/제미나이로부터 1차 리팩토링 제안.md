## 1. [높음] A2A Task 목록 직렬화/파싱의 예약어 충돌 및 에러 가능성
- **위치**: [mcp.rs:L190](file:///d:/privateProject/tunaRound/src/mcp.rs#L190) 및 [worker.rs:L58-107](file:///d:/privateProject/tunaRound/src/worker.rs#L58-L107)
- **분류**: 기능 안정성 / 코드 품질
- **결합 태그**: Functional
- **문제**: A2A Task 목록 데이터를 조립(`format_open_tasks`)하고 파싱(`parse_open_tasks`)할 때, `[id] from=... state=... ctx=... msg=...` 형태의 단순 문자열 템플릿과 `find` 메소드를 활용하고 있습니다. 만약 LLM의 지시문이나 대화 주제 본문(`msg`)에 ` state=`, ` msg=`, ` ctx=` 등의 파싱 예약어가 포함되어 있을 경우 파싱 오동작 및 데이터 누락이 발생합니다.
- **왜 문제인가**: 에이전트 간 주고받는 메시지 본문은 자유 형식이므로 예약어와 유사한 패턴이나 줄바꿈 등이 들어갈 가능성이 높습니다. 이로 인해 Task를 워커가 누락하거나 엉뚱한 정보로 잘못 인식하는 등 기능적인 오작동이 발생할 수 있습니다.
- **변경 전파 범위**:
  - 이 파일이 바뀌면 같이 바뀔 가능성이 있는 파일: [mcp.rs](file:///d:/privateProject/tunaRound/src/mcp.rs), [worker.rs](file:///d:/privateProject/tunaRound/src/worker.rs)
  - 전파 원인: 두 모듈이 단순 문자열 포맷(Ad-hoc string format)의 세부 명세에 의존하고 있습니다.
  - 결합 유형: Functional
- **제안**: Task 정보를 전달할 때 JSON/JSONL 형식의 구조화된 포맷을 활용하도록 A2A MCP 도구와 워커 파서를 개선하거나, `msg` 본문 영역을 인코딩(Base64 등)하여 전송할 것을 제안합니다.
- **리스크**: 포맷 변경 시 이전 버전 코어/워커 간의 하위 호환성이 깨질 수 있으므로 프로토콜 버저닝이 필요할 수 있습니다.
- **우선순위 근거**: 사용자가 자유롭게 입력할 수 있는 토론 주제 및 LLM 발언 내용에 따라 워커가 에러를 발생시키며 토론 진행이 완전히 멈출 수 있는 취약한 영역이므로 우선순위가 높습니다.

## 2. [중간] `exec.rs` 내 Watchdog의 PID 기반 강제 종료와 PID Recycling 레이스 리스크
- **위치**: [exec.rs:L112-132](file:///d:/privateProject/tunaRound/src/runner/exec.rs#L112-L132) 및 [exec.rs:L184-197](file:///d:/privateProject/tunaRound/src/runner/exec.rs#L184-L197)
- **분류**: 기능 안정성
- **결합 태그**: 해당 없음
- **문제**: `run_with_watchdog`의 타임아웃 감시 스레드가 프로세스를 강제 종료할 때, `std::process::Child::kill` 메소드를 사용하지 않고 OS 전역 셸 명령어(`kill -9` / `taskkill`)에 PID를 문자열로 넘겨 실행하는 `kill_pid`를 사용하고 있습니다.
- **왜 문제인가**: 프로세스 종료 감지와 `kill` 호출 간의 동기화가 완벽히 원자적이지 않습니다. 극히 낮은 확률이지만, 자식이 정상 수거된 직후 다른 엉뚱한 프로세스가 해당 PID를 빠르게 재할당받을 경우 그 무관한 시스템 프로세스를 종료시키는 위험(PID recycling race)이 존재합니다.
- **변경 전파 범위**:
  - 이 파일이 바뀌면 같이 바뀔 가능성이 있는 파일: [exec.rs](file:///d:/privateProject/tunaRound/src/runner/exec.rs)
  - 전파 원인: 내부 watchdog 스레드의 구현 방식에 국한됩니다.
  - 결합 유형: 해당 없음
- **제안**: watchdog 스레드가 PID를 기반으로 직접 셸 명령을 실행하여 킬하는 대신, `Child` 인스턴스의 소유권이나 OS 핸들을 스레드 간 안전하게 공유(`Arc<Mutex<Option<Child>>>` 등)하여 러스트 표준 라이브러리의 `Child::kill` API를 직접 호출하도록 구조를 개선할 것을 제안합니다.
- **리스크**: 자식 프로세스 핸들의 생명주기 제어 구조를 재설계해야 하므로 구현 변경 오버헤드가 발생합니다.
- **우선순위 근거**: 드물지만 서버 환경이나 타 시스템에서 예기치 않은 시스템 프로세스 킬로 이어질 수 있는 명확한 안정성 레이스 컨디션이므로 중간순위입니다.

## 3. [중간] Runner Trait의 동기 인터페이스와 OS 스레드 남용 구조
- **위치**: [runner/mod.rs:L47-49](file:///d:/privateProject/tunaRound/src/runner/mod.rs#L47-L49) 및 [worker.rs:L192-200](file:///d:/privateProject/tunaRound/src/worker.rs#L192-L200)
- **분류**: 경계와 거리 / 책임 분리 / 코드 품질
- **결합 태그**: Functional
- **문제**: `Runner` trait의 `run` 함수가 동기(Sync) 인터페이스로 설계되어 있어, 비동기 런타임 위에서 네트워크 요청(예: `reqwest::blocking`)이나 외부 프로세스 spawn을 수행할 때 스레드 블로킹을 해소하기 위해 `std::thread::spawn`을 통해 매번 OS 수준 스레드를 직접 띄우는 임시방편(Workaround)을 사용 중입니다.
- **왜 문제인가**: 비동기 런타임(`tokio`)을 적극 도입한 프로젝트임에도 동기 인터페이스 설계로 인해 OS 스레드를 수동 스폰하고 있으며, 이는 장기적으로 리소스 비효율을 증가시킵니다. 또한 `tokio::task::spawn_blocking`과의 호환성 문제로 인해 런타임 제어가 두 갈래로 나뉩니다.
- **변경 전파 범위**:
  - 이 파일이 바뀌면 같이 바뀔 가능성이 있는 파일: [runner/mod.rs](file:///d:/privateProject/tunaRound/src/runner/mod.rs) 및 이를 구현하는 모든 러너들 (`claude.rs`, `codex.rs`, `http.rs`, `opencode.rs`), 그리고 호출처인 `orchestrator/mod.rs`, `worker.rs`, `repl/mod.rs` 등
  - 전파 원인: 핵심 Runner 계약(`Runner` trait)이 동기이기 때문에 모든 구현체 및 실행 스레드 배선이 영향을 받습니다.
  - 결합 유형: Contract / Functional
- **제안**: `Runner` trait의 `run` 인터페이스를 비동기(`async fn run` 또는 `#[async_trait]`)로 변경하고, `reqwest` 비동기 클라이언트 및 비동기 process spawn을 사용하도록 변경할 것을 제안합니다.
- **리스크**: 거의 모든 러너와 실행 루프를 비동기 형태로 전면 리팩토링해야 하므로 수정 범위가 넓고 리스크가 큽니다.
- **우선순위 근거**: 당장 장애를 내지는 않으나, 비동기 조율이 핵심인 멀티세션 아키텍처에서 장기적인 성능 및 스레드 병목의 원인이 될 수 있는 중요한 구조 개선 항목입니다.

## 4. [중간] `main.rs`에서의 다중 SQLite Connection 생성 및 무거운 책임
- **위치**: [main.rs:L786-828](file:///d:/privateProject/tunaRound/src/main.rs#L786-L828), [main.rs:L921-931](file:///d:/privateProject/tunaRound/src/main.rs#L921-L931), [main.rs:L980-986](file:///d:/privateProject/tunaRound/src/main.rs#L980-L986), [main.rs:L1104-1117](file:///d:/privateProject/tunaRound/src/main.rs#L1104-L1117)
- **분류**: 경계와 거리 / 책임 분리
- **결합 태그**: Functional
- **문제**: `main.rs`에서 동일한 DB 파일 경로를 기반으로 `SqliteStore::open`을 Retriever, ValiditySink, CoreSync, A2A Store 등 여러 서브시스템 배선 시마다 개별 호출하여 독립된 SQLite Connection을 중복 생성하고 있습니다. 또한, `main.rs`가 1000라인이 넘는 긴 절차적 코드로 구현되어 있어 너무 많은 조립 책임을 담당합니다.
- **왜 문제인가**: 단일 프로세스 내에서 동일 SQLite 파일에 커넥션을 불필요하게 여러 개 생성할 경우 리소스 낭비 및 커넥션 락 경합 리스크가 있으며, 시스템 시작 시점 에러 제어가 한곳에 모이지 않고 흩어집니다.
- **변경 전파 범위**:
  - 이 파일이 바뀌면 같이 바뀔 가능성이 있는 파일: [main.rs](file:///d:/privateProject/tunaRound/src/main.rs) 및 [sqlite.rs](file:///d:/privateProject/tunaRound/src/store/sqlite.rs)
  - 전파 원인: `main.rs`에 직접 기술된 시스템 초기화 구조 때문입니다.
  - 결합 유형: Functional
- **제안**: 스토어 커넥션 생성을 단일화하고 `Arc` 등을 활용해 하위 서비스들에 안전하게 공유 배선하거나, 스토어 빌더/팩토리 책임을 별도 모듈로 분리할 것을 제안합니다.
- **리스크**: `SqliteStore`의 `Send`/`Sync` 관련 Mutex 래핑 배선 작업이 수반되어야 합니다.
- **우선순위 근거**: 당장 SQLite는 WAL 및 내부 락 시스템 덕분에 작동되나, 초기화 과정에서의 구조적 복잡성과 연결 남발로 인한 비효율성을 정리할 필요가 있습니다.

- **왜 문제인가**: `TUNAROUND_EMBED_MODEL` 환경 변수를 통해 사용자가 1024차원이 아닌 다른 차원 수(예: 768차원인 `nomic-embed-text` 등)를 지닌 Ollama 임베딩 모델을 주입해 사용하는 경우, DB 저장 시 벡터 길이 불일치(`blob.len() != dim * 4`) 검증 조건 등으로 인해 임베딩 데이터가 누락되거나 무시될 위험이 있습니다.
- **변경 전파 범위**:
  - 이 파일이 바뀌면 같이 바뀔 가능성이 있는 파일: [embedding.rs](file:///d:/privateProject/tunaRound/src/store/embedding.rs)
  - 전파 원인: 임베더 내부 구현에 국한됩니다.
  - 결합 유형: 해당 없음
- **제안**: 최초 임베딩 결과(1회 호출 시점)나 Ollama API 메타데이터 조회를 통해 동적으로 실제 임베딩 차원을 구하거나, 설정 파일에서 `dim`을 지정받을 수 있도록 개선을 제안합니다.
- **리스크**: 거의 없음 (임베더 모듈 내부의 국소적인 개선).
- **우선순위 근거**: 사용자가 기본 모델 외의 다른 임베딩 모델로 교체하고자 할 때 발생할 수 있는 에지 케이스이나, 현재는 기본 `qwen3-embedding` 모델(1024차원)을 주로 쓰고 있으므로 우선순위는 낮습니다.
