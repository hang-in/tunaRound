# tunaRound (A): 상주 코어 + 접속 클라이언트

> 사용자 제기(2026-06-30): "코어(오케스트레이션+검색/메모리)를 백엔드 서비스로 상주시키고 claude code·codex·ollama/openai 클라이언트가 접속해서 쓰는 게 낫지 않나?" + "자율 루프는 사용자가 명시적으로 켤 때만 가치. 아직은 사람의 마이크로매니징 = 품질업·비용다운."
>
> 이 문서는 [v2-context-memory-direction](v2-context-memory-direction_2026-06-30.md) line 73 백로그("분리 터미널 A2A 자율 협업")를 본격 설계로 승격한다. 검색/메모리 스택은 그 문서가 정본, 여기서는 그 위의 **프로세스/통제 구조**만 다룬다.

## 1. 결론

A2A에는 성격이 다른 둘이 섞여 있다. 가치는 (A)에 있고 (B)는 조건부다.

- **(A) 아키텍처 A2A** = 코어를 상주 백엔드로, 에이전트를 접속 클라이언트로. 사람이 운전자. `--mcp-search`(Plan 14)가 깔아둔 씨앗의 확장. **= 이 문서의 대상.**
- **(B) 자율 A2A** = 에이전트가 다음 화자를 스스로 정하고 서로 트리거. 분산 turn-triggering 난제이자, 사람-주도 thesis와 충돌. **= 미래 옵션, 지금 구현 X.**

설계가 지켜야 할 단 하나: **turn-control을 명시적 정책(turn-policy)으로 분리**해 (B)를 포크가 아닌 플러그인으로 남겨둔다.

## 2. 왜 (A)가 가치인가 (측정 가능한 형태)

지금은 매 라운드 tunaRound가 에이전트를 stateless spawn(`-p`/`exec`/HTTP)하고 **맥락을 prompt에 push**한다. `--recent-turns`(Plan 16)는 그 push량을 줄이는 우회였다. (A)의 본질은 push를 pull로 뒤집는 것이다.

- **영속 코어**: 전사·검색·요약·로스터를 1급 서비스로 노출. 멀티 프론트/멀티세션(Redis 씨앗, Plan 06)이 여기서 완성.
- **클라이언트가 맥락을 pull**: 매 턴 전체 재전송 대신 필요한 슬라이스만 도구로 당김. 재주입량 감소를 **실측**(Stage 2의 통과 기준).
- **클라이언트 플러그인화**: 모델/러너 추가 = "백엔드 프로토콜 구현".

## 3. 왜 (B)는 미래 옵션인가 (경제 논리)

자율 루프가 비싼 이유는 토큰이 아니라 **탐색 공간**이다. 사람이 마이크로매니징하면 매 턴 **가지치기**가 일어나 틀린 방향이 즉시 제거되고 라운드 수가 준다. 이것이 품질↑·비용↓의 진짜 원천이다. 자율 루프는 이 가지치기를 잃어 수렴까지 **헛도는 라운드**에 토큰을 태운다.

(B)의 경제가 뒤집히는 조건은 둘 중 하나다.

1. 토큰 단가가 충분히 떨어져 헛도는 라운드 비용이 무시 가능.
2. 과제가 **검증 가능**(테스트/컴파일/실측이 옳고 그름을 기계 판정)해서 사람 없이 루프가 스스로 수렴.

그래서 (B)는 "사용자가 명시적으로 켜는, 조건부로만 경제적인 옵션"이다. 디폴트는 영원히 사람-주도.

## 4. 핵심 추상: turn-policy 솔기

"다음 턴을 누가 정하는가"를 코어의 명시적 정책으로 뽑는다.

```
trait TurnPolicy {
    // 다음 화자(또는 종료)를 결정. 전사/로스터/사람 입력을 본다.
    fn next_speaker(&mut self, ctx: &SessionView, human_input: Option<Directive>) -> TurnDecision;
}

enum TurnDecision { Speak(SeatId), AwaitHuman, Conclude }
```

- **HumanDriven** (디폴트, 유일 구현): 사람이 `@claude`로 지목할 때만 `Speak`, 그 외 `AwaitHuman`. = 현행 동작 그대로.
- **AutoLoop** (미래, (B)): 규칙/에이전트가 다음 화자 결정. 같은 백엔드(`read_transcript`/`post_turn`) 위에 **정책만 교체**. 사용자가 명시 opt-in으로만 활성.

이 솔기 하나가 "지금은 사람 운전 + 나중에 자율 루프"를 동시에 만족시킨다. (B)를 켜기 전엔 비용 0.

## 5. 단계적 롤아웃 (각 단계 단독 가치, 과설계 경계)

| Stage | 내용 | 통과 기준 | 위험 |
|---|---|---|---|
| **0** | 코어 서비스 경화: 검색 품질(항목1) + 요약 carry-forward(항목2) | 검색 품질 게이지 개선 실측 / 요약이 라운드 간 이월 | 낮음 (기존 모듈 개선) |
| **1** | 오케스트레이션 툴 확장: 기존 rmcp 서버에 `read_transcript(since_cursor)`·`get_roster` (+선택 `post_turn`) 추가 | claude·codex가 MCP 클라이언트로 `read_transcript` 실호출 | codex MCP 지원 범위 미검증 |
| **2** | 주입 모델 전환(crux): 전체 push 대신 얇은 포인터 주입 + 에이전트 pull | 재전송 토큰량 감소 실측 vs 현행 | "그 턴에 뭘 봤나" 통제 약화 |
| **3** | 코어 프로세스 분리: 상주 데몬 + 프론트/에이전트 각각 클라이언트 | 멀티 프론트 동시 접속 / 멀티세션 | 프로세스 수명·동시성 |
| **4** | (범위 밖) 영속 에이전트 세션 + AutoLoop 정책 = (B) | 경제 조건(3절) 입증 시에만 | 자율 thesis 충돌 |

원칙: Stage 1~2는 에이전트를 **여전히 stateless spawn**하되 맥락만 pull로 받게 한다(저위험). 영속 프로세스는 Stage 3 이후 - (B) 자율과 경계가 가까우니 신중.

## 6. 리스크 / 미리 못 박을 것

- **claude/codex의 MCP 클라이언트 능력**: 우리 코어를 MCP 서버로 물고 `read_transcript`를 실제 호출하는지 = Stage 1 통과 기준. codex의 MCP 지원 범위 확인 필요(Plan 14 Task 4에서 `-c mcp_servers` 인자 수용은 확인됨, 실 도구 호출은 미검증).
- **Stage 2 함정**: pull로 넘기면 정밀 통제가 약해질 수 있음. 포인터에 **당길 수 있는 범위를 명시**해 통제 유지. 재주입축소/RAG의 강점을 잃지 않게.
- **stateless vs 영속**: Stage 1~2 저위험 유지가 핵심. 영속 프로세스로 점프하지 않는다.
- **A2A 표준**: Google Agent2Agent 정렬은 후속. 로컬 MCP + 버스로 시작.

### 배포 토폴로지 (사용자 확정 2026-06-30): 코어는 얇다

코어 = 프로토콜 + 상태(전사·FTS·벡터BLOB·Redis핫·오케스트레이션). **추론기 아님** - 무거운 연산(LLM)은 전부 클라이언트. 임베딩 계산도 이미 원격 Ollama 오프로드라 코어는 코사인만. 따라서 CPU/메모리 가벼운 조정 계층 = 어디든 호스팅 가능.

- **토폴로지**: (a) 개인 서버 상주, (b) LAN 피어(로컬 한 대가 호스트), (c) 외부 SaaS. 연산은 자유.
- **비용은 신뢰 경계에서 발생**: 코어가 로컬 신뢰 경계를 벗어날수록 (1) 인증/신원 (2) 격리/멀티테넌시 (3) 데이터 거주성(전사 = 설계논의+레포맥락; 외부 호스팅 = 그 맥락 송신, "서비스 비공개" 기조와 충돌 가능) 필요.
- **단계 영향**: Stage 0~2는 전부 로컬이라 무관. 토폴로지/인증은 Stage 3에서 본격.

### Stage 3 메모: 데이터 계층 + WAN 연결 (사용자 결정 2026-06-30, 지금은 로컬로 둠)

배포 모델 = **자체호스팅 오픈소스**(각자 받아 자기 서버/로컬에 구동), **SaaS 아님**. → 단일 테넌트라 멀티테넌시·SaaS 인증 부담 없음.

- **데이터 계층**: 배포별(=사용자별) **SQLite 파일**(WAL + busy_timeout) + **단일 Redis**(user/session 키 네임스페이스). PostgreSQL 비채택: 10명 규모 이득<비용, 사용자별 파일이면 쓰기 경합 없음·격리 공짜, 한국어 FTS5+벡터 스택 재작성 회피([[prefer-sqlite-over-vector-db]]). store는 트레잇 뒤(SqliteStore/Retriever/Indexer/TranscriptReader)라 미래 PG는 구현 교체로 가능(YAGNI, 지금 안 함).
- **임베딩 엔드포인트**: 현재 원격 Ollama(bge-m3)는 실사용상 **사용자 소유·상시 가용**(이전 필요시 조치 가능, 사용자 확인 2026-06-30). SaaS 의존 리스크 아님. 별도 전용 endpoint는 규모 커질 때만.
- **WAN 연결 = 두 관계 분리**:
  - **(a) 코어 → WAN LLM**: 코어가 클라이언트 = outbound API 호출 = 포워딩/터널 불필요(공개 엔드포인트·API key면). 현 SSH 터널은 친구 Ollama가 localhost+무인증이라 생긴 특수케이스.
  - **(b) 원격 에이전트 → 내 코어(분산 A2A)**: 코어가 서버 = inbound = NAT 문제 본체. **해법 = 포트포워딩 대신 오버레이(Tailscale/WireGuard)**: 각 머신 tailnet 가입, 사설 IP 통신, NAT 자동통과, 공개포트 0 → WAN을 "LAN 피어"로 되돌림. (대안: cloudflared/ngrok outbound 리버스 터널, 단 인증 필수.)
- **코드 무관**: 도달성은 transport/운영 선택(코어는 주소 바인딩 + 프로토콜만). Stage 1-2(로컬) 차단 안 함.
