# tunaRound v1 설계 기록

> 이 문서는 **현재 tunaRound 전체 구조의 설계 명세가 아닙니다.**
>
> 2026년 6월 29일, tunaRound가 Claude Code와 Codex를 연결하는 사람 주도 설계 토론 REPL로 시작할 때 확정한 v1 설계를 요약한 역사 문서입니다. 이후 세션 분기, 검색과 기억, 원격 코어, A2A 작업 위임, 워커 노드, presence scanner, Codex relay, 웹 대시보드가 추가되면서 프로젝트 범위가 크게 넓어졌습니다.

## 1. 당시의 한 줄 정의

터미널에서 사용자가 운전하는 Claude Code와 Codex가 같은 레포를 읽고 설계를 토론한 뒤, 결론을 문서로 남기는 도구였습니다.

핵심은 자동으로 끝까지 토론하는 swarm이 아니라 **사람이 방향을 정하는 3자 대화**였습니다.

```text
사용자
├─ Claude Code: 제안자
└─ Codex: 리뷰어
```

## 2. 왜 만들었나

초기 문제는 다음과 같았습니다.

- 구현 전에 여러 관점으로 설계를 검토하고 싶었습니다.
- 데스크톱 앱 전체를 띄우지 않고 터미널에서 가볍게 사용하고 싶었습니다.
- Claude Code와 Codex가 같은 레포를 직접 읽게 하고 싶었습니다.
- 토론 결론을 복사해 다시 정리하는 일을 줄이고 싶었습니다.
- 사람이 관찰자가 아니라 토론의 운전자가 되어야 했습니다.

이 기준에 따라 tunaFlow와 tunapi에서 검증된 토론·러너 구조를 작은 Rust CLI로 옮기는 것이 v1의 목표였습니다.

## 3. 초기 구조

v1은 네 부분으로 나뉘었습니다.

| 구성 요소 | 책임 |
| --- | --- |
| 에이전트 러너 | Claude Code와 Codex CLI 실행, 스트림 파싱, 오류 처리 |
| 토론 오케스트레이터 | 역할 주입, 순차 응답, 지목, 후속 질문 |
| 전사와 결과 문서 | 대화 저장과 결론 기록 |
| REPL | 사용자 입력과 응답 출력 |

코어 로직이 REPL이나 특정 transport를 직접 의존하지 않게 하는 원칙은 이후 구조에서도 유지되었습니다.

```text
REPL
 ↓
오케스트레이터
 ↓
러너와 도메인 타입
```

## 4. 당시의 핵심 결정

### 사람이 운전한다

사용자가 메시지를 보내면 에이전트가 응답하고, 사용자가 다음 질문으로 방향을 조정합니다. 에이전트끼리 사람 없이 계속 토론하지 않습니다.

### 같은 레포를 직접 읽는다

긴 컨텍스트를 별도 패키지로 복사하지 않고, 각 CLI가 현재 작업 디렉터리의 레포를 직접 읽습니다.

### 역할을 분리한다

한 자리는 제안자, 다른 자리는 리뷰어처럼 역할을 주어 같은 자료를 다른 관점에서 보게 합니다.

### 순차적으로 응답한다

뒤에 응답하는 자리는 앞 에이전트의 같은 라운드 답변을 볼 수 있습니다.

```text
사용자 질문
  ↓
Claude 응답
  ↓
Codex가 Claude 응답까지 보고 답변
```

### 쓰기는 명시적으로 지목한다

기본 토론은 읽기 중심으로 진행하고, 실제 파일 변경은 사용자가 특정 에이전트에 명시적으로 맡기는 구조였습니다.

### 종료는 사람이 결정한다

자동 수렴 점수로 끝내지 않고, 사용자가 충분하다고 판단할 때 결론을 정리합니다.

## 5. v1의 작업 흐름

```text
1. 사용자가 질문한다.
2. 오케스트레이터가 역할과 이전 대화를 포함한 프롬프트를 만든다.
3. 첫 번째 에이전트가 응답한다.
4. 두 번째 에이전트가 앞 응답을 참고해 답한다.
5. 두 응답을 전사에 기록한다.
6. 사용자가 다음 질문이나 지목으로 방향을 정한다.
7. 사용자가 종료하면 결론을 문서로 남긴다.
```

## 6. v1에서 의도적으로 제외했던 것

초기 범위를 작게 유지하기 위해 다음 기능은 v1에서 제외했습니다.

- 세 개 이상의 자유로운 좌석 구성
- 여러 머신의 세션 공유
- 작업 브로커와 헤드리스 워커
- 웹 대시보드
- 라이브 세션 자동 발견
- A2A 프로토콜 기반 작업 위임
- 리치 TUI

이 기능들은 이후 실제 필요가 확인되면서 순차적으로 추가되었습니다.

## 7. 이후 무엇이 달라졌나

| v1 당시 | 현재 |
| --- | --- |
| Claude와 Codex 두 자리 | Claude, Codex, OpenCode, HTTP LLM, 외부 A2A 에이전트 |
| 로컬 REPL | 로컬 REPL, 원격 코어, 웹 대시보드 |
| 단일 머신 | LAN과 여러 머신의 mesh |
| 토론 중심 | 토론, 관측, 검색, 작업 위임, 결과 회수 |
| 파일 중심 전사 | SQLite 기반 전사, 검색 색인, 작업 장부 |
| 사람이 세션을 직접 찾음 | presence scanner가 라이브 세션을 로스터에 동기화 |
| Codex 수신 경로 없음 | codex-relay가 라이브 thread에 작업 전달 |

다만 다음 원칙은 유지되었습니다.

- 사람이 목표와 최종 판단을 맡습니다.
- 에이전트는 역할과 작업 단위로 참여합니다.
- 완전 자율 swarm은 목표가 아닙니다.
- 코어와 프런트엔드의 경계를 분리합니다.
- 실제 결과와 결정이 레포와 장부에 남아야 합니다.

## 8. 이 문서를 남기는 이유

현재 코드의 사용법이나 운영 방법을 설명하기 위해서가 아니라, tunaRound가 왜 사용자 주도 구조를 고수하는지 보여주기 위해 남깁니다.

현행 기능과 명령을 판단할 때는 이 문서보다 README, reference 문서, 현재 코드와 테스트를 우선합니다.
